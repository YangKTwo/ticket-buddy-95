## 1. 从 Lovable 到 Vercel 的完整流程

### 流程概览

```
Lovable 生成项目 → 推送到 GitHub → 本地 clone/pull → 配置 .env
    → 修改代码/部署配置 → push → Vercel 关联 GitHub → 自动构建部署
```

### 各步骤说明

| 步骤 | 在做什么 | 说明 |
|------|----------|------|
| **1. Lovable 构建项目** | 在线生成 TanStack Start 应用 | 生成页面、路由、Supabase 集成等；在 Lovable 沙箱中会自动启用 Nitro。克隆到本机或 Vercel 后需自行保证构建配置正确。 |
| **2. 推送到 GitHub** | 把代码同步到远程仓库 | 作为唯一源码来源；支持版本管理；供本地与 Vercel 拉取同一份代码。 |
| **3. 拉到本地** | `git clone` / `git pull` | 在本地开发、调试；配置不提交到 Git 的 `.env`；修改 `vite.config.ts`、`vercel.json` 等后再 push。 |
| **4. 配置环境变量** | 编辑根目录 `.env` | 仅影响本地 `npm run dev`；**不会**随 Git 上传，线上需在 Vercel 控制台单独配置。 |
| **5. 提交并 push** | `git add` → `commit` → `push` | 将部署修复与业务改动同步到 GitHub，触发 Vercel 构建（若已关联）。 |
| **6. Vercel 关联 GitHub** | Import 仓库 | 建立项目与仓库绑定；监听分支 push；在 Vercel 配置 Environment Variables。 |
| **7. 部署** | 云端 `npm run build` → 发布 | Install → Build → 将 `.vercel/output` 发布到 CDN + Serverless。 |

### 一次 Vercel 部署内部发生的事

1. **Install**：`npm install`（根据 `package-lock.json` 安装依赖）
2. **Build**：`npm run build`
   - 客户端资源 → `.vercel/output/static/`
   - 服务端函数 → `.vercel/output/functions/__server.func/`
   - 路由配置 → `.vercel/output/config.json`
3. **Deploy**：将 `.vercel/output` 交给 Vercel 边缘网络，绑定域名

**触发方式**：push 到已关联分支（自动），或在 Dashboard 中 **Redeploy**（改环境变量后常用）。

### 各角色一句话

| 环节 | 作用 |
|------|------|
| Lovable | 生成并在线预览项目 |
| GitHub | 存代码，连接本地与 Vercel |
| 本地 + `.env` | 开发调试，密钥不进 Git |
| push | 把正确构建配置交给 GitHub |
| Vercel ↔ GitHub | 有 push 就自动构建上线 |
| Vercel 环境变量 | 线上构建与运行用的密钥 |
| 部署 | 云端构建并发布 `.vercel/output` |

## 2. Vercel 404 NOT_FOUND 问题

### 问题现象

部署显示成功，但访问站点出现平台级错误页：

- **404: NOT_FOUND**
- **Code:** `NOT_FOUND`
- 带部署/请求 ID（如 `sfo1::...`、`hkg1::...`）

这是 **Vercel 平台级** 404（找不到有效路由/函数），不是应用内的业务 404 页面。

### 错误原因（两层）

#### 原因 1：非 Lovable 环境未启用 Nitro

项目使用 `@lovable.dev/vite-tanstack-config`。在 **Vercel CI** 等非 Lovable 环境中，默认会 **跳过 Nitro 插件**，仅产出：

- `dist/client`（静态资源，无根目录 `index.html`）
- `dist/server`

缺少 Vercel 所需的 Serverless 函数与 `config.json` 路由，被当作不完整静态站 → 404。

构建日志可能出现：

```
No Lovable context detected — skipping nitro deploy plugin.
Pass `nitro: true` (or `nitro: { ... }`) to force-enable.
```

**修复**：在 `vite.config.ts` 中显式启用 `nitro: { preset: "vercel" }`。

#### 原因 2：Nitro 输出目录不符合 Vercel 规范

即便启用了 `preset: "vercel"`，Lovable 默认仍输出到 `dist/server/`，而路由指向 **`/__server`**。Nitro 官方 Vercel 结构应为：

```
.vercel/output/
  ├── config.json          # 路由：静态 + 其余 → /__server
  ├── static/              # 前端 JS/CSS 等
  └── functions/
      └── __server.func/   # SSR 服务端函数（与 /__server 对应）
      └── index.mjs
```

`dist/server` 与 `/__server` 不匹配 → Vercel 找不到函数 → 继续 NOT_FOUND。

**修复**：在 `vite.config.ts` 中覆盖 `output` 为 Nitro 标准路径，并新增 `vercel.json` 指定 `outputDirectory`。

#### 原因 3：误用 SPA 式 vercel.json（曾出现）

曾配置将路由重写到 `index.html` 的 `vercel.json`。TanStack Start 是 **SSR**，没有根目录 `index.html`，该配置不适用，已删除。

### 解决方案汇总

| 改动 | 作用 |
|------|------|
| `vite.config.ts`：`nitro.preset: "vercel"` + 标准 `output` 路径 | 强制启用 Nitro，输出到 `.vercel/output/functions/__server.func` |
| `vercel.json`：`outputDirectory: ".vercel/output"`，`framework: null` | 使用 Build Output API，避免按纯 Vite 静态站处理 |
| `.gitignore`：忽略 `.vercel/output` | 不将构建产物提交进仓库 |
| Vercel 环境变量 | 与 404 无关，但运行 Supabase/API 需要 |

### 修复后的 vite.config.ts 要点

```ts
nitro: {
  preset: "vercel",
  output: {
    dir: ".vercel/output",
    serverDir: ".vercel/output/functions/__server.func",
    publicDir: ".vercel/output/static",
  },
},
```

### 修复后的 vercel.json

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "buildCommand": "npm run build",
  "installCommand": "npm install",
  "framework": null,
  "outputDirectory": ".vercel/output"
}
```

### 不要做的事

- 不要将 Output Directory 设为 `dist` 或 `dist/client`
- 不要添加 SPA 式 `rewrites` 到 `index.html`
- Root Directory 保持 `./` 即可（代码在仓库根目录）

## 3. Vercel 项目设置参考

以下设置验证可用，一般无需修改：

| 设置项 | 推荐值 |
|--------|--------|
| **Root Directory** | `./`（仓库根即项目根） |
| **Framework Preset** | Other（或由 `vercel.json` 处理） |
| **Build Command** | `npm run build` |
| **Output Directory** | `.vercel/output` |
| **Install Command** | `npm install` |

**Root Directory** 仅在使用 monorepo、代码在子目录（如 `apps/web`）时才需修改；与本项目无关。

## 4. 环境变量配置

### 本地 `.env`（不提交 Git）

用于 `npm run dev`，示例：

```env
SUPABASE_PUBLISHABLE_KEY="..."
SUPABASE_URL="https://xxx.supabase.co"
VITE_SUPABASE_PROJECT_ID="xxx"
VITE_SUPABASE_PUBLISHABLE_KEY="..."
VITE_SUPABASE_URL="https://xxx.supabase.co"
VITE_QWEN_API_KEY="..."
```

### Vercel 控制台（线上）

路径：**Project → Settings → Environment Variables**

勾选 **Production** 与 **Preview**，重新部署后生效。

| 变量 | 是否建议配置 | 用途 |
|------|--------------|------|
| `VITE_SUPABASE_URL` | **要** | 浏览器端 Supabase（构建时注入） |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | **要** | 同上 |
| `SUPABASE_URL` | **要** | 服务端 SSR / 中间件 `process.env` |
| `SUPABASE_PUBLISHABLE_KEY` | **要** | 同上 |
| `VITE_SUPABASE_PROJECT_ID` | 可选 | 当前代码未引用 |
| `VITE_QWEN_API_KEY` | 需要 AI 时配置 | AI 服务 |
| `SUPABASE_SERVICE_ROLE_KEY` | 若用管理员客户端 | `client.server.ts` 需要时配置 |

### 为何 VITE_* 与 SUPABASE_* 两套都要？

- 客户端（`client.ts`）读 `import.meta.env.VITE_*`（构建时打进前端）
- 服务端（`auth-middleware.ts` 等）读 `process.env.SUPABASE_*`（运行时）

在 Vercel 上 **VITE_*** 需在构建时存在；**SUPABASE_*** 需在运行时存在。值可与本地 `.env` 相同。

## 5. 后端部署在哪里

本项目 **没有** 单独一台 24 小时运行的传统后端服务器，而是 **三块** 组合：

### 5.1 应用服务端 — Vercel Serverless

| 项 | 说明 |
|----|------|
| **位置** | `.vercel/output/functions/__server.func/` |
| **技术** | TanStack Start + Nitro 构建的 Serverless 函数 |
| **职责** | SSR 渲染、`.server.ts` 逻辑、`src/server.ts` 入口、鉴权中间件、Server Functions 等 |
| **触发** | 用户访问页面时按请求执行，非常驻进程 |

同一次 `npm run build` 同时产出：

- **前端静态资源** → `.vercel/output/static/`
- **服务端函数** → `.vercel/output/functions/__server.func/`

### 5.2 数据与认证 — Supabase（独立云服务）

| 项 | 说明 |
|----|------|
| **位置** | 不在 Vercel，在 Supabase 项目 URL（如 `*.supabase.co`） |
| **职责** | PostgreSQL 数据库、Auth 登录、REST/Realtime API |
| **调用方** | 浏览器或 Vercel 上的 Serverless 通过 HTTPS 访问 |

```
用户浏览器 → Vercel（页面 + SSR 函数）
                 │
                 └──→ Supabase（数据、登录）
```

### 5.3 第三方 API（可选）

如 `VITE_QWEN_API_KEY`：请求发往对应 API 服务商，密钥仅配置在 Vercel 环境变量中。

### 5.4 vercel.json 的角色

`vercel.json` **不是**后端本身，只告诉 Vercel 如何构建、从哪个目录读取产物（含 static + `__server.func`）。

### 5.5 与传统全栈对比

| 传统全栈 | 本项目 |
|----------|--------|
| 前端：Nginx 静态站 | Vercel `static/` |
| 后端：Node 常驻服务器 | Vercel `__server.func` |
| 数据库：自建 MySQL | Supabase 托管 |
| 多次独立部署 | push → Vercel 自动构建；数据库在 Supabase |

**一句话**：页面与服务端逻辑在 **Vercel**；数据库与登录在 **Supabase**。

## 6. 项目中的关键文件

| 文件 | 作用 |
|------|------|
| `vite.config.ts` | Vite + TanStack Start + Nitro（Vercel preset 与输出路径） |

| `package.json` | 依赖与 `npm run build` 脚本 |
